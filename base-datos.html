Módulo 2: Conexión a Base de Datos Segura

📌 ¿Por qué no usar consultas directas?

Es tentador escribir $conexion->query("SELECT * FROM usuarios WHERE id = $id"). Pero si un usuario malintencionado escribe 1 OR 1=1 en la URL, tu base de datos le entregará toda la tabla. Esto se llama SQL Injection.

2.1 La analogía del "Sobre Sellado"

✅ Consultas Preparadas (Prepared Statements)

Imagina que envías una carta. En lugar de escribir la dirección dentro del mensaje (donde alguien podría alterarla), escribes la dirección en el sobre. El cartero (la base de datos) sabe que el sobre es solo para dirigir el paquete, no es parte del mensaje.

Las Consultas Preparadas hacen exactamente eso: separan la estructura de la consulta de los datos. La base de datos compila la estructura primero, y luego inserta los datos como texto literal, imposibilitando que se ejecuten como código.

2.2 Implementación con PDO y Patrón Singleton

✅ Código profesional reutilizable

Usamos el patrón Singleton para asegurar que solo se abra una conexión a la base de datos en toda la aplicación, ahorrando memoria.

namespace App\Models;
use PDO;

class Database {
    private static ?self $instance = null;
    private PDO $connection;

    private function __construct() {
        $dsn = "mysql:host=localhost;dbname=mi_bd;charset=utf8mb4";
        $options = [
            PDO::ATTR_ERRMODE => PDO::ERRMODE_EXCEPTION,
            PDO::ATTR_EMULATE_PREPARES => false, // ⚠️ CRÍTICO: Fuerza a la BD a preparar, no a PHP
        ];
        $this->connection = new PDO($dsn, "root", "", $options);
    }

    public static function getInstance(): self {
        if (self::$instance === null) {
            self::$instance = new self();
        }
        return self::$instance;
    }

    public function getConnection(): PDO {
        return $this->connection;
    }
}

2.3 Transacciones: "Todo o Nada"

✅ Evita datos a medias

Si necesitas guardar un usuario Y su perfil, y la segunda consulta falla, no quieres que el usuario quede guardado sin perfil. Las transacciones lo evitan.

$pdo = Database::getInstance()->getConnection();
try {
    $pdo->beginTransaction(); // Inicia la "grabación"
    
    $pdo->prepare("INSERT INTO usuarios (email) VALUES (?)")->execute([$email]);
    $pdo->prepare("INSERT INTO perfiles (user_id) VALUES (?)")->execute([$nuevo_id]);
    
    $pdo->commit(); // ¡Éxito! Guarda todo permanentemente
} catch (Exception $e) {
    $pdo->rollBack(); // Algo falló: deshace TODO como si nada hubiera pasado
    error_log("Error: " . $e->getMessage());
}

💡 Regla de Oro para tu proyecto

NUNCA concatenes variables directamente en una consulta SQL. Siempre usa prepare() y execute(). Si sigues esta regla, tu proyecto será inmune a SQL Injection.
