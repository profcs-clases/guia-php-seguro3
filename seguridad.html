Módulo 3: Autenticación y Seguridad Web

📌 La seguridad no es un extra, es la base

Un sistema de login no es solo "verificar si la contraseña coincide". Implica proteger la sesión, evitar que roben identidades y garantizar que, incluso si hackean la base de datos, las contraseñas sean inútiles para el atacante.

3.1 Hashing: La bóveda de contraseñas

❌ Error común: Guardar contraseñas en texto plano o con MD5

Si guardas "123456" tal cual, y un hacker roba tu base de datos, tiene todas las contraseñas. MD5 y SHA1 ya están obsoletos y se pueden descifrar en segundos con computadoras modernas.

✅ Solución: Hashing adaptativo (Bcrypt / Argon2)

El hashing es como una licuadora: puedes convertir una fruta en jugo, pero no puedes convertir el jugo de nuevo en fruta. PHP lo hace fácil y seguro, agregando automáticamente una "sal" (salt) aleatoria.

// 1. AL REGISTRAR: Hashear la contraseña
$hash = password_hash($_POST['password'], PASSWORD_DEFAULT); 
// Guarda $hash en la BD (se verá algo como: $2y$10$e0... )

// 2. AL INICIAR SESIÓN: Verificar
$stmt = $pdo->prepare("SELECT id, password_hash FROM usuarios WHERE email = ?");
$stmt->execute([$_POST['email']]);
$user = $stmt->fetch();

if ($user && password_verify($_POST['password'], $user['password_hash'])) {
    // ¡Contraseña correcta!
    session_regenerate_id(true); // 🛡️ Previene que roben tu sesión
    $_SESSION['user_id'] = $user['id'];
}

3.2 Protección CSRF (Cross-Site Request Forgery)

✅ El "Secreto Compartido"

Un atacante podría engañar a tu navegador para que envíe un formulario sin que lo sepas. La defensa es un Token CSRF: un código secreto que solo tu servidor y tu formulario conocen.

// 1. Generar token al iniciar la sesión
if (empty($_SESSION['csrf_token'])) {
    $_SESSION['csrf_token'] = bin2hex(random_bytes(32));
}

// 2. Incluirlo en el formulario HTML
<input type="hidden" name="csrf_token" value="<?= $_SESSION['csrf_token'] ?>">

// 3. Validar al recibir el POST
if ($_POST['csrf_token'] !== $_SESSION['csrf_token']) {
    die("Error de seguridad: Solicitud no autorizada.");
}

3.3 Prevención de XSS (Cross-Site Scripting)

✅ Escapar al mostrar

Si un usuario escribe <script>alert('hack')</script> en su perfil, y tú lo muestras tal cual, ese código se ejecutará en el navegador de cualquiera que visite su perfil. La solución es simple:

// SIEMPRE usa htmlspecialchars al mostrar datos del usuario en HTML
echo "Hola, " . htmlspecialchars($nombre_usuario, ENT_QUOTES, 'UTF-8');

💡 Checklist de Sesiones Seguras

  • ✅ Configura session.cookie_httponly = 1 (evita que JavaScript robe la cookie).
  • ✅ Usa session_regenerate_id(true) después de cada login o cambio de rol.
  • ✅ Al hacer logout, usa session_destroy() y borra la cookie.
  • ✅ Nunca le digas al usuario "El email no existe". Di siempre "Credenciales inválidas" para no regalar información.
