Módulo 4: Arquitectura, Validación y Buenas Prácticas

📌 La regla de oro: "Nunca confíes en el cliente"

Cualquier dato que venga del navegador (formularios, URLs, cookies) puede haber sido modificado. La validación en JavaScript es solo para mejorar la experiencia del usuario; la validación real debe ocurrir siempre en el servidor (PHP).

4.1 La analogía de la Línea de Ensamblaje

✅ Flujo de una petición segura

Imagina una fábrica. Un producto (la petición del usuario) pasa por varias estaciones de control antes de ser aprobado:

  1. Middleware (Seguridad): ¿El usuario tiene permiso para estar aquí?
  2. Validador (Calidad): ¿Los datos tienen el formato correcto (email válido, cédula correcta)?
  3. Controlador (Gerente): Coordina el proceso.
  4. Modelo (Almacén): Guarda los datos limpios en la base de datos.
  5. Vista (Empaque): Muestra el resultado al usuario.

📊 Diagrama: Flujo de Validación

4.2 Clase Validadora Reutilizable

✅ No repitas código de validación

En lugar de hacer 10 if (empty(...)), crea una clase que acumule los errores.

class Validator {
    private static array $errors = [];

    public static function required(string $field, mixed $value): bool {
        if (empty(trim((string)$value))) {
            self::$errors[$field] = "El campo $field es obligatorio.";
            return false;
        }
        return true;
    }

    public static function email(string $field, string $value): bool {
        if (!filter_var($value, FILTER_VALIDATE_EMAIL)) {
            self::$errors[$field] = "Correo electrónico inválido.";
            return false;
        }
        return true;
    }

    public static function getErrors(): array { return self::$errors; }
    public static function isValid(): bool { return empty(self::$errors); }
}

Uso en el controlador:

Validator::required('email', $_POST['email']);
Validator::email('email', $_POST['email']);

if (!Validator::isValid()) {
    $_SESSION['errors'] = Validator::getErrors();
    header("Location: /registro"); // Vuelve al formulario para mostrar errores
    exit;
}

4.3 Depuración Profesional (Adiós al var_dump)

❌ Error común en producción

Dejar var_dump($usuario); die(); o tener display_errors = On en un servidor real. Esto le regala a los hackers la estructura de tu base de datos y rutas de archivos.

✅ Logging silencioso

// En config.php (Producción)
ini_set('display_errors', 0); // No mostrar en pantalla
ini_set('log_errors', 1);     // Guardar en archivo
ini_set('error_log', __DIR__ . '/../storage/logs/errores.log');

// En tu código:
try {
    // Código que puede fallar
} catch (Exception $e) {
    error_log("Fallo en registro: " . $e->getMessage());
    echo "Ocurrió un error inesperado. Intente más tarde."; // Mensaje amigable
}

💡 Consejos para tu proyecto final

  • 🎯 Usa htmlspecialchars() en cada variable que imprimas en HTML.
  • 🎯 Para subir archivos, no confíes en $_FILES['archivo']['type']. Usa finfo_file() para verificar el tipo real del archivo y guárdalo con un nombre aleatorio (ej: uniqid() . '.jpg').
  • 📚 Para investigar después: Aprende a usar Composer (gestor de dependencias) y Variables de Entorno (.env) para no subir contraseñas a GitHub.
